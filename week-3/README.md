# Week 3 - RAG Indexing Pipeline

# 3주차 이론 과제: RAG의 구성 요소

I. 이론
 
## 1. RAG란?

RAG(Retrieval-Augmented Generation)는 대규모 언어모델이 답변을 생성할 때, 모델 내부에 학습된 지식만 사용하는 것이 아니라 외부 문서 저장소에서 관련 정보를 검색한 뒤, 그 검색 결과를 바탕으로 답변을 생성하는 방식이다. 원 논문에서는 이를 파라메트릭 메모리(모델 내부 지식)와 논파라메트릭 메모리(외부 검색 인덱스)를 결합하는 구조로 설명한다.

RAG가 필요한 이유는 다음과 같다.  
첫째, 모델이 학습하지 못한 최신 정보나 특정 도메인 문서를 반영할 수 있다.  
둘째, 모델이 근거 없이 답을 만들어내는 할루시네이션을 줄이는 데 도움이 된다.  
셋째, 모델 자체를 다시 학습하지 않고도 외부 문서를 교체하거나 확장할 수 있다.

2주차 방식처럼 system prompt에 전체 데이터를 통째로 삽입하는 방식은 문서가 짧을 때는 단순하지만, 문서가 길어질수록 컨텍스트 길이 한계와 비용 문제가 커지고, 질문과 무관한 내용까지 함께 들어가 비효율적이다. 반면 RAG는 문서를 미리 청크 단위로 나누고 벡터화해 두었다가, 질문이 들어오면 관련 청크만 검색하여 넣기 때문에 더 확장 가능하고 실무적이다. 

RAG 파이프라인의 전체 흐름은 다음과 같다.

- **Indexing 단계**: 문서 로딩 → 청크 분할 → 임베딩 생성 → 벡터 저장소 저장
- **Retrieval/Generation 단계**: 질문 입력 → 질문 임베딩 → 유사 청크 검색 → 검색된 컨텍스트 기반 답변 생성

즉 RAG는 “문서를 미리 검색 가능한 구조로 만들어 두고, 질문 시 관련 근거만 꺼내어 답변 생성에 활용하는 방식”이라고 정리할 수 있다.
---

## 2. RAG에서 사용되는 용어

### Chunking
긴 문서를 검색 가능한 작은 단위로 나누는 과정이다.  
`chunk_size`는 한 청크의 길이이고, `chunk_overlap`은 인접 청크 간 중복 길이로, 문맥 단절을 줄이는 역할을 한다. 

### Embedding
텍스트를 숫자 벡터로 변환하는 과정이다.  
의미가 비슷한 문장일수록 벡터 공간에서 가깝게 배치되어 유사도 검색이 가능해진다. OpenAI의 임베딩 문서도 임베딩을 검색, 클러스터링, 추천 등에 활용할 수 있다고 설명한다. 

### Vector Store
임베딩 벡터를 저장하고 유사도 검색을 수행하는 저장소이다.  
대표적으로 FAISS와 Chroma가 있으며, RAG에서 검색 속도와 검색 정확도를 좌우하는 핵심 구성 요소다.

### Retriever
사용자의 질문과 의미적으로 가까운 청크를 찾아오는 컴포넌트이다.  
보통 질문도 임베딩한 뒤, 벡터 저장소에서 가장 가까운 청크를 Top-K 방식으로 검색한다. 여기서 Top-K는 가장 유사한 결과를 몇 개까지 가져올지를 의미한다. 

### Generation
검색된 청크를 컨텍스트로 사용하여 LLM이 최종 답변을 생성하는 단계이다.  
즉 retrieval이 근거를 찾는 과정이라면, generation은 그 근거를 바탕으로 자연어 답변을 정리하는 과정이다. 

---

## 3. 외부 데이터 소스 기반 응답 생성

LLM 단독 응답은 모델 내부에 학습된 지식만 기반으로 답변을 생성한다. 이 방식은 빠르지만, 최신 정보나 특정 문서의 세부 내용에는 약할 수 있다. 반면 외부 데이터를 검색하여 응답하는 방식은 질문과 관련된 문서를 먼저 회수한 뒤, 해당 근거를 바탕으로 답을 만들기 때문에 더 높은 정확도와 근거성을 기대할 수 있다.

일반적인 API 호출 흐름은 다음과 같다.

질문 입력 → 질문 임베딩 생성 → 벡터 검색 → 관련 청크 회수 → 컨텍스트 구성 → LLM 호출 → 답변 생성.

RAG는 할루시네이션을 완전히 제거하는 것은 아니지만, 모델이 아무 근거 없이 추측하는 대신 외부 문서를 참조하게 함으로써 답변을 근거에 묶어둘 수 있다. 따라서 RAG의 핵심 역할은 단순 검색이 아니라, **답변을 grounding하는 구조를 제공하는 것**이라고 볼 수 있다. 다만 검색 품질이 낮거나 잘못된 청크를 회수하면 오히려 부정확한 답변이 생성될 수도 있으므로, retrieval 단계가 매우 중요하다.

---

## 4. LangChain 기반 RAG 파이프라인 구조

LangChain에서 RAG 파이프라인은 일반적으로 다음 요소들로 구성된다.

| 단계 | LangChain 컴포넌트 |
|---|---|
| 문서 로딩 | Document Loader (`PyPDFLoader` 등) |
| 청킹 | Text Splitter (`RecursiveCharacterTextSplitter` 등) |
| 임베딩 | Embeddings (`OpenAIEmbeddings`, `HuggingFaceEmbeddings` 등) |
| 벡터 저장 | Vector Store (`FAISS`, `Chroma` 등) |
| 검색 + 생성 | Retriever + Chain |

이 구조의 장점은 각 단계를 독립적인 컴포넌트로 교체할 수 있다는 점이다. 예를 들어 PDF 로더를 다른 로더로 바꾸거나, splitter 전략을 바꾸거나, vector store를 FAISS에서 Chroma로 교체하는 식의 실험이 가능하다. 그래서 LangChain은 RAG를 빠르게 프로토타이핑하고 비교 실험하기에 적합한 프레임워크라고 볼 수 있다.

## 5. 실습 과제 예측 (실습 전 가설)

### (1) 의료급여 PDF를 청킹할 때 가장 큰 어려움
의료급여 PDF는 일반 서술형 문서가 아니라, 제도 설명, 표, FAQ(Q/A), 예외 조건이 섞여 있다. 따라서 단순 문자 수 기준으로 청킹할 경우 질문과 답변, 혹은 본문과 예외 조건이 서로 다른 청크로 분리될 가능성이 높다. 특히 FAQ 문서는 질문과 답변을 한 덩어리로 유지하는 것이 중요한데, 청킹 경계가 잘못 잡히면 검색 시 질문은 맞게 검색되더라도 답변 핵심 문장이 포함되지 않을 수 있다고 예상하였다.

### (2) Golden Dataset 5문제 중 어떤 난이도의 질문에서 검색 실패가 많을지 예측
단순 사실형 질문(easy)은 상대적으로 검색 성공률이 높을 것으로 예상하였다. 반면 조건이 여러 개 붙거나 예외 규정이 함께 포함된 medium~hard 질문에서는 검색 실패가 더 많을 것으로 예상하였다. 그 이유는 이러한 질문이 한 문장만 찾는 문제가 아니라, 서로 연결된 조건과 답변을 함께 회수해야 하기 때문이다.

## 6. 실습 후 결과 비교

실제로 검색 평가를 수행한 결과, 총 5문항 중 4문항이 HIT, 1문항이 FAIL로 나타나 검색 성공률은 80%였다. 실패한 문항은 Q3(등록 장애인의 의료급여의뢰서 필요 여부)였으며, 관련 질문이 포함된 청크 자체는 Top-1에서 검색되었지만, 정답 판정에 필요한 핵심 답변 문장 전체가 동일 청크 안에 충분히 포함되지 않아 FAIL로 판정되었다.

즉 실습 전 가설대로, 단순 fact형 질문보다 **질문-답변-예외 조건이 함께 유지되어야 하는 문항에서 청킹 경계 문제가 실제로 드러났다**. 따라서 실습 전 예측은 대체로 맞았다고 볼 수 있다. 이번 결과를 통해 FAQ형 문서는 단순 character-based chunking보다, 질문/답변 블록 단위의 구조적 청킹이 더 적합할 가능성이 높다는 점을 확인하였다.

## 1. 과제 개요

이번 과제에서는 `2024 알기 쉬운 의료급여제도.pdf` 문서를 대상으로 RAG(Retrieval-Augmented Generation)용 인덱싱 파이프라인을 구축하고, 직접 만든 Golden Dataset을 기반으로 검색 성능을 평가하였다.

이번 단계의 핵심은 생성형 답변을 만드는 것보다, **문서를 검색 가능한 형태로 가공하고 질문에 대해 적절한 근거 청크를 회수할 수 있는지 검증하는 것**에 있다.

## 2. 사용 문서

- 문서명: `2024 알기 쉬운 의료급여제도.pdf`
- 문서 형식: PDF
- 문서 내용: 의료급여 제도 개요, 절차, 본인부담금, 선택의료급여기관 이용절차, FAQ 등

이 문서는 제도 설명, 표, FAQ(Q/A) 형식이 혼합되어 있어 일반 문단형 텍스트보다 청킹 전략의 영향을 크게 받는 문서라고 판단하였다.

## 3. Golden Dataset 구성

Golden Dataset은 문서를 직접 읽고 사람이 정답과 근거를 확인하여 작성하였다.  
총 5개의 질문을 만들었으며, 난이도는 easy 2개, medium 2개, hard 1개로 구성하였다.

### Golden Dataset 설계 기준
- 단순 사실형 질문
- 예외 규정이 포함된 질문
- 조건 분기형 질문
- FAQ 답변의 핵심 의미를 반영해야 하는 질문

### 구성한 질문 예시
- 의료급여의뢰서 제출 기한
- 15세 이하 아동의 예외 적용 여부
- 등록 장애인의 의료급여의뢰서 필요 여부
- 선택의료급여기관 재의뢰 가능 여부
- 2종수급권자 15세 이하 아동의 입원 본인부담 적용

## 5. RAG Indexing Pipeline 구현

이번 과제에서는 Python과 LangChain 기반으로 인덱싱 파이프라인을 구현하였다.

### 파이프라인 흐름
1. PDF 로딩
2. 페이지 단위 문서 추출
3. 텍스트 청킹
4. 각 청크에 대한 임베딩 생성
5. FAISS 벡터스토어에 저장
6. 검색 평가 시 질문을 임베딩하여 Top-k 유사 청크 검색

### 사용한 설정
- Chunk Size: `800`
- Chunk Overlap: `150`
- Top-k: `3`

### Chunking 전략 선택 이유
의료급여 PDF는 다음 요소가 함께 포함되어 있다.

- FAQ 질문/답변 블록
- 표 형식 정보
- 예외사항과 조건문
- 제도 설명 문단

따라서 너무 큰 청크는 검색 정밀도를 떨어뜨릴 수 있고, 너무 작은 청크는 질문과 답변이 분리될 위험이 있다. 이를 고려하여 중간 크기의 청크(`800`)와 overlap(`150`)을 적용하였다.

## 7. 검색 평가 방법

검색 평가는 Golden Dataset의 각 질문을 입력으로 사용하여, 벡터스토어에서 Top-3 유사 청크를 검색하는 방식으로 수행하였다.

초기에는 `evidence_text`의 완전 문자열 포함 여부로 hit를 판정하였으나, PDF 텍스트 추출 과정에서 줄바꿈과 공백 차이로 인해 실제 관련 청크를 검색했음에도 False가 발생하였다. 이를 해결하기 위해 공백/줄바꿈 정규화 후 evidence 포함 여부를 판정하도록 수정하였다.

검색 평가의 상세 결과는 `retrieval_eval_results.json` 파일에 별도로 저장하였다.

## 9. 실패 원인 분석

### Q3 실패 원인
Q3는 “등록 장애인이 제2차의료급여기관에서 진료를 받을 때 반드시 제1차의료급여기관의 의료급여의뢰서가 필요한가요?”에 대한 질문이었다.

검색 결과를 보면 관련 질문(Q15)이 포함된 청크 자체는 Top-1에서 회수되었다. 그러나 정답 판정에 필요한 핵심 답변 문장 전체가 동일 청크 안에 충분히 포함되지 않아 evidence hit가 False로 판정되었다.

즉 이 실패는 검색 자체가 완전히 잘못된 것이 아니라,
- 질문과 답변이 서로 다른 chunk에 걸렸거나
- FAQ 문답 블록이 청킹 경계에서 분리되었기 때문으로 해석할 수 있다.

## 10. 개선 방향

이번 결과를 바탕으로 다음과 같은 개선을 고려할 수 있다.

1. `chunk_size`를 소폭 증가시켜 질문과 답변이 함께 유지되도록 조정
2. `chunk_overlap`을 증가시켜 청크 경계 손실을 줄이기
3. FAQ 문서를 일반 문자 길이 기준이 아니라 `Q/A` 블록 단위로 우선 분할
4. Top-k를 3에서 5로 늘려 회수율 비교
5. 평가 시 evidence 전체 문자열 완전일치뿐 아니라 키워드 기반 보조 판정 도입

## 11. 금지사항 관련 정리

본 과제는 ChatGPT/Claude 웹 UI를 사용하지 않고, Python 환경에서 직접 구현하였다.

PDF 로딩, 청킹, 임베딩, 벡터 인덱싱, 검색 평가를 모두 코드로 수행하였다. 임베딩 모델을 사용한 벡터 유사도 검색만 수행하였다.

## 12. 느낀 점

이번 과제를 통해 RAG에서 중요한 것은 단순히 문서를 벡터화하는 것만이 아니라, **문서 구조에 맞는 청킹 전략과 적절한 평가 방법을 함께 설계하는 것**이라는 점을 확인할 수 있었다.

특히 PDF 기반 FAQ 문서는 질문과 답변, 예외조건이 서로 밀접하게 연결되어 있기 때문에, 문서 구조를 고려하지 않은 단순 문자 기반 청킹은 retrieval 성능 저하로 이어질 수 있다는 점을 확인하였다.
