I. 프롬프트 기법 4가지 정리

## 1. Zero-shot Prompting
### 한 줄 정의
Zero-shot prompting은 **예시를 주지 않고 작업 지시만으로 모델이 답하게 하는 프롬프트 방식**이다. ([promptingguide.ai][1])

### 핵심 원리
LLM은 기본적으로 앞에 나온 토큰들을 바탕으로 다음 토큰을 예측하는 **autoregressive 생성 모델**이다.
Zero-shot에서는 예시 없이도 “이 입력은 어떤 작업인가”를 **명령문 자체**로 알려 준다. 그러면 모델은 사전학습과 instruction tuning 과정에서 학습한 패턴을 바탕으로, 주어진 지시와 가장 잘 맞는 출력 분포를 따라 답을 생성한다. Prompt Engineering Guide도 오늘날의 모델이 대규모 학습과 지시 따르기 튜닝 덕분에 zero-shot 작업 수행이 가능하다고 설명한다. ([promptingguide.ai][1])
즉, zero-shot의 핵심은
**예시 없이도 지시문이 출력 방향을 정렬해 준다**는 데 있다.
다만 예시가 없기 때문에, 모델이 작업 형식이나 세부 규칙을 스스로 추정해야 해서 복잡한 과제에서는 흔들릴 수 있다.

### 사용 예시
가이드에서는 감정 분류 예시를 든다.
예를 들어 “텍스트를 중립, 부정 또는 긍정으로 분류합니다. 
텍스트: 휴가는 괜찮을 것 같아요. 감정:”처럼 입력하면, 모델은 별도 예시 없이도 이를 감정 분류 문제로 해석하고 “중립”이라고 답한다. 
즉, **문제 유형과 출력 형식이 비교적 명확하면 예시 없이도 수행 가능하다**는 예다. ([promptingguide.ai][1])

### 장점
* 프롬프트가 짧고 간단하다.
* 빠르게 baseline을 만들기 좋다.
* 분류, 요약, 번역처럼 작업 형식이 분명한 문제에 유리하다.

### 단점
* 출력 형식이 불안정할 수 있다.
* 세부 조건이 많은 문제에서는 규칙을 놓치기 쉽다.
* 계산, 다단계 추론, 예외 처리 문제에서는 오답 가능성이 커진다.

### 언제 쓰면 좋은지 / 언제 효과가 적은지
**좋은 경우**
* 작업 목적이 명확할 때
* 출력 형식이 단순할 때
* 우선 baseline 성능을 보고 싶을 때

**효과가 적은 경우**
* 조건이 여러 개 겹치는 문제
* 표 해석, 계산, 논리 추론처럼 중간 판단이 필요한 문제
* 출력 포맷을 엄격하게 맞춰야 하는 문제


## 2. Few-shot Prompting
### 한 줄 정의
Few-shot prompting은 **입력과 출력의 예시 몇 개를 먼저 보여 준 뒤, 같은 패턴으로 새 문제를 풀게 하는 프롬프트 방식**이다. ([promptingguide.ai][2])

### 핵심 원리
Autoregressive 모델은 앞에 놓인 텍스트 패턴을 강하게 따른다.
Few-shot은 프롬프트 안에 **데모(example)** 를 넣어서, 모델이 “아, 지금 해야 하는 작업은 이런 형식이고 이런 방식으로 답하면 되는구나”를 문맥 안에서 바로 추론하게 만든다. 
가이드도 few-shot을 “데모를 제공하여 모델이 더 나은 성능을 발휘하도록 유도하는 문맥 내 학습”이라고 설명한다. ([promptingguide.ai][2])
즉, few-shot의 효과는
**예시가 다음 토큰 예측의 기준 틀을 만들어 주기 때문**이다.
모델은 단순히 문제를 푸는 것이 아니라, 방금 본 입력-출력 짝의 패턴을 이어서 완성하려고 한다.

### 사용 예시
가이드에서는 새로운 단어를 문장에 쓰는 예시를 보여 준다.
예를 들어 “whatpu”라는 낯선 단어가 들어간 예문을 먼저 하나 보여 주고, 이어서 “farduddle”이라는 단어에 대해 문장을 만들라고 하면, 모델은 앞선 예문 형식을 따라 적절한 문장을 생성한다. 즉, **예시 하나만으로도 모델이 요구된 작업 방식을 흉내 내며 일반화할 수 있다**는 것을 보여 준다. ([promptingguide.ai][2])
또한 가이드는 라벨이 완전히 정확하지 않아도, **형식 자체가 유지되는 것만으로도 도움이 될 수 있다**고 설명한다. 이는 few-shot이 단순 정답 암기보다 **패턴 유도 효과**가 크다는 뜻이다. ([promptingguide.ai][2])

### 장점
* 출력 형식을 안정시키기 좋다.
* zero-shot보다 작업 의도를 더 정확히 전달할 수 있다.
* 새로운 태스크나 낯선 형식에 빠르게 적응시키기 좋다.

### 단점
* 예시를 잘못 고르면 오히려 성능이 떨어질 수 있다.
* 프롬프트 길이가 길어진다.
* 복잡한 추론 문제에서는 예시 몇 개만으로 한계가 있다. 가이드도 few-shot이 복잡한 추론에는 완전한 해결책이 아니라고 설명한다. ([promptingguide.ai][2])

### 언제 쓰면 좋은지 / 언제 효과가 적은지
**좋은 경우**
* 원하는 출력 형식을 분명히 보여 주고 싶을 때
* 규칙 기반 분류, 정보 추출, 포맷 변환처럼 패턴 모방이 중요한 문제
* zero-shot에서 답변 형식이 흔들릴 때

**효과가 적은 경우**
* 계산이나 논리 전개가 핵심인 복잡한 추론 문제
* 예시를 충분히 넣기 어려운 긴 문서 처리
* 예시 분포가 실제 문제와 너무 다를 때

## 3. Chain-of-Thought (CoT)
### 한 줄 정의
Chain-of-Thought prompting은 **최종 답만 말하게 하지 않고, 중간 추론 단계를 함께 생성하도록 유도하는 방식**이다. ([promptingguide.ai][3])

### 핵심 원리
Autoregressive 모델은 한 번에 정답 전체를 떠올리는 것이 아니라, 토큰을 순차적으로 생성한다.
CoT는 이 순차 생성 특성을 이용해 **정답 이전에 중간 판단 과정**을 먼저 펼치게 만든다. 
그러면 모델은 문제를 여러 작은 단계로 나누어 처리하면서, 각 단계의 텍스트가 다음 단계의 조건이 된다. 결과적으로 복잡한 문제를 곧바로 찍는 대신, **추론 경로를 따라가며 정답에 도달할 확률**이 높아진다.
가이드도 CoT가 “중간 추론 단계를 통해 복잡한 추론을 가능하게 한다”고 설명하며, 단답형 프롬프트보다 복잡한 작업에서 더 나은 결과를 준다고 말한다. ([promptingguide.ai][3])

### 사용 예시
가이드의 대표 예시는 홀수 합 문제다.
단순히 “이 집합의 홀수를 다 더하면 짝수인가?”라고 물으면 모델이 틀릴 수 있는데, CoT 예시에서는 먼저
* 홀수가 무엇인지 고르고
* 그 수들을 더하고
* 합이 짝수인지 홀수인지 판별한 뒤
* 최종적으로 참/거짓을 말한다.

이런 중간 단계를 포함하자 모델이 정답에 도달한다. ([promptingguide.ai][3])
또 가이드는 zero-shot CoT 예시로 “단계별로 생각해 보자”라는 한 문장을 추가했을 때, 사과 개수 계산 문제의 오답이 정답으로 바뀌는 사례도 보여 준다. 즉, **추론 경로를 말하게 하는 것 자체가 성능을 끌어올릴 수 있다**는 것이다. ([promptingguide.ai][3])

### 장점
* 계산, 논리, 다단계 추론에서 효과적이다.
* 모델이 중간 조건을 놓치는 일을 줄여 준다.
* 오답 분석이 쉬워진다. 어디서 틀렸는지 경로를 볼 수 있기 때문이다.

### 단점
* 출력이 길어진다.
* 항상 좋은 것은 아니다. 단순 작업에는 오히려 불필요할 수 있다.
* 그럴듯한 추론 과정을 써도 최종 답이 틀릴 수 있다.

### 언제 쓰면 좋은지 / 언제 효과가 적은지
**좋은 경우**
* 수학, 계산, 규칙 적용, 다중 조건 판별
* 표 해석 후 단계별 결론을 내려야 하는 문제
* intermediate reasoning이 중요한 문제

**효과가 적은 경우**
* 단순 분류, 번역, 짧은 요약처럼 바로 답해도 되는 문제
* 모델이 충분히 작은 경우
* 중간 추론이 길어지면서 오히려 잡음이 늘어나는 경우

## 4. Self-Consistency
### 한 줄 정의
Self-Consistency는 **한 번의 추론 결과만 믿지 않고, CoT 기반으로 여러 추론 경로를 생성한 뒤 가장 일관되게 나오는 최종 답을 선택하는 방식**이다. ([promptingguide.ai][4])

### 핵심 원리
Autoregressive 모델은 같은 문제라도 샘플링 방식에 따라 서로 다른 추론 경로를 만들 수 있다.
Self-Consistency는 바로 이 점을 활용한다. 하나의 탐욕적(greedy) 추론만 쓰는 대신, 여러 reasoning path를 생성하고 **가장 자주 또는 가장 일관되게 도달하는 답**을 채택한다. 가이드는 이를 CoT의 일반적인 greedy decoding을 대체하는 접근으로 설명한다. ([promptingguide.ai][4])
핵심은 **개별 경로는 틀릴 수 있지만, 올바른 답은 여러 독립적 추론 경로에서 반복적으로 수렴할 가능성이 높다**는 가정이다.
즉, 단일 생성의 우연한 오류를 줄이는 일종의 추론 앙상블이라고 볼 수 있다.

### 사용 예시
가이드에서는 나이 문제를 예로 든다.
“내가 6살일 때 여동생은 내 나이의 절반이었다. 지금 내가 70살이면 여동생은 몇 살인가?”라는 질문에 단순 출력은 35라고 틀리게 답할 수 있다. 
하지만 self-consistency는 CoT 예시를 바탕으로 여러 추론 경로를 생성하고, 그중에서 가장 일관된 답을 선택해 정확도를 높이려는 접근이다. 가이드는 이 방법이 산술 및 상식 추론 작업에서 CoT 성능을 향상시킬 수 있다고 설명한다. ([promptingguide.ai][4])

### 장점
* CoT보다 더 안정적인 정답을 얻을 수 있다.
* 한 번의 우연한 잘못된 추론을 줄여 준다.
* 산술, 상식, 복합 추론 문제에서 유리하다.

### 단점
* 여러 번 생성해야 해서 비용과 시간이 더 든다.
* 단순 문제에는 과하다.
* 잘못된 reasoning path가 반복되면 다수결이어도 틀릴 수 있다.

### 언제 쓰면 좋은지 / 언제 효과가 적은지
**좋은 경우**
* CoT는 유효하지만 답이 들쭉날쭉할 때
* 계산 문제, 복합 추론 문제
* 정확도가 중요하고 여러 번 호출 비용을 감수할 수 있을 때

**효과가 적은 경우**
* 단순 요약, 번역, 감정분석 같은 저난도 작업
* 프롬프트 비용이나 응답 시간이 제한적일 때
* 애초에 모델이 문제 구조를 잘못 이해하는 경우



### 한 번에 비교하면
* **Zero-shot**: 예시 없이 바로 시킴
* **Few-shot**: 예시를 보여 주고 따라 하게 함
* **CoT**: 생각 과정을 말하게 함
* **Self-Consistency**: 생각 과정을 여러 번 만들고 가장 일관된 답을 고름






II. 추가 기법

## 2-1. Generate Knowledge Prompting

### 한 줄 정의
모델이 문제를 풀기 전에 관련된 지식을 먼저 생성하도록 유도한 뒤, 그 지식을 기반으로 답을 도출하게 하는 기법이다.

### 핵심 원리
LLM은 autoregressive 방식으로 이전 토큰을 기반으로 다음 토큰을 생성한다.
이때 문제를 바로 풀게 하면 필요한 배경지식이 충분히 활성화되지 않을 수 있다.
Generate Knowledge Prompting은 먼저 관련 지식을 생성하게 하여 **추론에 필요한 context를 선행적으로 구성**하고, 이후 답을 생성하도록 만든다.

즉,👉 **지식 생성 → 추론 → 정답 도출**의 구조를 만든다.

### 사용 예시
```text
Q: 왜 하늘은 파란색인가?

먼저 이 질문과 관련된 과학적 사실들을 설명하라.
그 다음 최종 답을 작성하라.
```

### 장점
* 모델이 놓칠 수 있는 배경지식을 보완한다
* 복잡한 개념 문제에서 정확도를 높인다

### 단점
* 불필요한 지식 생성으로 노이즈가 생길 수 있다
* 잘못된 지식을 생성하면 오히려 성능이 떨어진다

### 언제 효과적인가 / 한계
* ✔️ 상식, 과학, 설명 기반 문제
* ❌ 단순 분류, 포맷 변환 문제

## 2-2. Prompt Chaining
### 한 줄 정의
하나의 프롬프트로 해결하지 않고, 여러 단계의 프롬프트를 순차적으로 연결하여 문제를 해결하는 방식이다.

### 핵심 원리
LLM은 한 번의 생성으로 모든 문제를 해결하려 하면 복잡한 작업에서 오류가 발생하기 쉽다.
Prompt Chaining은 문제를 여러 단계로 나누어 각각 처리하게 함으로써 **단계별 오류를 줄이고 전체 정확도를 높인다**.

즉,👉 **작업 분해 → 단계별 해결 → 결과 결합**

### 사용 예시
1단계:

```text
다음 문장에서 핵심 정보를 추출하라.
```

2단계:
```text
추출된 정보를 바탕으로 요약을 작성하라.
```

### 장점
* 복잡한 작업을 안정적으로 처리 가능
* 각 단계별 디버깅이 가능

### 단점
* 구현이 번거롭고 호출 비용 증가
* 단계 간 오류 전파 가능

### 언제 효과적인가 / 한계
* ✔️ 정보 추출 + 요약, 분석 + 판단 같은 복합 작업
* ❌ 단순 질문 응답

## 2-3. Tree of Thoughts (ToT)
### 한 줄 정의
여러 개의 추론 경로를 트리 구조로 탐색하고, 그 중 가장 적절한 경로를 선택하여 문제를 해결하는 기법이다.

### 핵심 원리
기존 CoT는 하나의 추론 경로만 생성한다.
하지만 ToT는 여러 reasoning path를 생성하고, 각 경로를 평가하면서 탐색한다.

즉,👉 **여러 사고 경로 생성 → 평가 → 최적 경로 선택**
이는 일종의 탐색 알고리즘(DFS/BFS)처럼 작동하며,
단일 경로 의존성을 줄인다.

### 사용 예시
```text
문제를 해결하기 위한 가능한 방법들을 여러 개 제시하라.
각 방법을 평가하고 가장 적절한 방법을 선택하라.
```

### 장점
* 복잡한 문제에서 높은 탐색 능력
* 다양한 해결 전략 고려 가능

### 단점
* 비용이 매우 높음
* 구현 난이도 높음

### 언제 효과적인가 / 한계

* ✔️ 퍼즐, 계획, 전략 문제
* ❌ 간단한 질문


## 2-4. ReAct (Reason + Act)
### 한 줄 정의
모델이 추론(Reasoning)과 행동(Action)을 번갈아 수행하면서 문제를 해결하는 방식이다.

### 핵심 원리
LLM은 단순히 텍스트 생성만 하는 것이 아니라, 외부 도구(API, 검색 등)와 결합할 수 있다.
ReAct는 **생각 → 행동 → 관찰 → 다시 생각**의 루프를 반복한다.

즉,👉 **Reason → Act → Observe → Reason 반복 구조**

### 사용 예시
```text
Question: 서울의 현재 날씨는?

Thought: 현재 날씨 정보가 필요하다
Action: weather_api("Seoul")
Observation: 10도, 흐림
Thought: 정보를 얻었다
Final Answer: 현재 서울은 10도이며 흐린 날씨입니다.
```

### 장점
* 외부 정보 활용 가능
* 현실적인 문제 해결 능력 향상

### 단점
* 구현 복잡
* 잘못된 행동 선택 가능

### 언제 효과적인가 / 한계
* ✔️ 실시간 정보, 검색, API 연동 문제
* ❌ 순수 텍스트 문제


## 2-5. Meta Prompting
### 한 줄 정의
프롬프트 자체를 모델이 생성하거나 개선하도록 하여 성능을 향상시키는 기법이다.

### 핵심 원리
LLM은 텍스트 생성 능력을 활용하여 **더 좋은 프롬프트를 스스로 생성**할 수 있다.
이를 통해 사람이 직접 설계하지 않아도 최적의 프롬프트를 찾을 수 있다.

즉,👉 **프롬프트 생성 → 평가 → 개선 반복**

### 사용 예시
```text
이 문제를 가장 잘 해결할 수 있는 프롬프트를 작성하라.
```

### 장점
* 프롬프트 설계 자동화
* 성능 최적화 가능

### 단점
* 결과가 불안정
* 평가 기준이 필요

### 언제 효과적인가 / 한계
* ✔️ 프롬프트 최적화, 자동화
* ❌ 단순 문제

---

## 🔥 한 줄 정리
* Generate Knowledge → **지식 먼저 생성**
* Prompt Chaining → **단계 분해**
* ToT → **여러 경로 탐색**
* ReAct → **추론 + 행동 루프**
* Meta Prompting → **프롬프트 자체를 생성**





III. 실습 과제 예측

의료급여 본인부담률 자료와 문제 데이터를 분석한 결과,
해당 과제는 단순 분류가 아니라 **조건 조합 + 규칙 매칭 + 일부 계산**이 결합된 문제로 판단된다.
이에 따라 프롬프트 기법별 성능 차이가 발생할 것으로 예상된다.

---

## 3-1. Zero-shot 정답률 예측
### 예측
Zero-shot의 정답률은 **낮거나 중간 수준(약 40~60%)**일 것으로 예상한다.

### 이유
Zero-shot은 예시 없이 작업을 수행하기 때문에 모델이 다음과 같은 문제를 겪을 가능성이 있다.

* 의료급여 표의 **복잡한 조건 구조를 정확히 파악하지 못함**
* 수급권자 유형, 나이, 의료기관 종류 등의 **다중 조건을 동시에 고려하지 못함**
* 일부 문제에서 요구되는 **계산 과정 누락 또는 오류 발생**

특히 이 과제는 단순한 텍스트 이해가 아니라
👉 **조건 기반 규칙 적용 문제**에 가깝기 때문에
모델이 규칙을 안정적으로 적용하지 못할 가능성이 높다.


## 3-2. 가장 큰 개선이 일어날 난이도 예측
### 예측
가장 큰 개선은 **medium 난이도에서 발생**할 것으로 예상한다.

### 이유
* **easy 문제**
  * 단일 조건 또는 명확한 규칙 적용
  * Zero-shot에서도 어느 정도 해결 가능
  * → 개선 폭이 제한적

* **medium 문제**
  * 여러 조건이 결합되지만 구조가 완전히 복잡하지는 않음
  * Few-shot 또는 CoT 적용 시 규칙 적용이 안정화됨
  * → **가장 큰 성능 향상 기대**

* **hard 문제**
  * 조건 조합 + 계산 + 복수 답변 요구
  * CoT로 개선되지만 여전히 계산 오류나 누락 가능성 존재
  * → 개선은 있지만 한계 존재

즉,
👉 medium은 “프롬프트 개선 효과가 가장 잘 드러나는 구간”이다.

## 3-3. 가장 큰 정답률 점프 구간 예측
### 예측
가장 큰 성능 향상은
👉 **Zero-shot → Few-shot 또는 Few-shot → CoT 구간에서 발생**할 것으로 예상한다.

### (1) Zero-shot → Few-shot
**이유**
* Few-shot은 입력-출력 예시를 제공하여
  * 출력 형식 안정화
  * 규칙 적용 방식 학습
    을 가능하게 한다.

따라서 모델이 문제를 어떻게 풀어야 하는지 명확히 이해하게 되어
👉 **초기 성능 개선이 크게 발생할 가능성**

### (2) Few-shot → CoT
**이유**
* Few-shot은 패턴은 잡아주지만, 복잡한 문제에서는 여전히 한계가 있다
* CoT는 추론 과정을 단계적으로 수행하게 만들어

  * 조건 누락 감소
  * 계산 정확도 향상
    을 기대할 수 있다

👉 따라서 두 번째 큰 점프 구간이 될 가능성 높음

### (3) CoT → Self-Consistency
**예상**
* 성능 향상은 있지만 상대적으로 작을 것으로 예상
* 
**이유**
* Self-Consistency는 이미 생성된 추론을 여러 번 반복하여 안정화하는 단계
* 근본적인 reasoning 개선보다는 **오답 보정 역할**

## 🔥 전체 흐름 정리

| 단계               | 예상 효과         |
| ---------------- | ------------- |
| Zero-shot        | baseline, 불안정 |
| Few-shot         | 형식 및 규칙 이해 개선 |
| CoT              | 추론 안정성 크게 향상  |
| Self-Consistency | 최종 정확도 보정     |

## ✔️ 최종 가설 요약

* Zero-shot 정답률: **낮거나 중간 수준**
* 가장 큰 개선 난이도: **medium**
* 가장 큰 점프 구간:
  👉 **Zero-shot → Few-shot / Few-shot → CoT**
