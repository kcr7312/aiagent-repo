# 1주차 과제: LLM 호출을 테스트하고 이해하기

1주차는 Agent를 만들지 않습니다. 1주차의 목표는 LLM API를 직접 호출하고, 요청/응답 구조와 structured output을 이해하는 것입니다. LLM 호출을 이해하고 초기 환경 구성을 확인합니다.

## 과제 목표

이번 주에는 고객 문의 티켓 12건을 LLM으로 구조화 추출합니다.

1. LLM 호출 시 `system` / `user` 메시지를 어떻게 나누는지
2. `model`, `temperature`, `max_tokens`를 왜 그렇게 설정했는지
3. 응답을 자유 텍스트가 아니라 `고정 JSON 스키마`로 강제하는 이유는 무엇인지
4. 모델 응답을 코드에서 어떻게 파싱하고 검증하는지
5. 프롬프트 v1과 v2를 바꾸면 `파싱 성공률`과 `정확도`가 어떻게 달라지는지

해당 프롬프트 모범 사례를 참고하여 여러 번 테스트를 수행하고 각 테스트 시 결과가 어떻게 변경되는지 확인해보시는 것을 권장드립니다.

## 공통 태스크

입력:
- 고객 문의 티켓 텍스트 12건

출력:
- 아래 JSON 스키마를 만족하는 구조화 결과

```json
{
  "intent": "order_change | shipping_issue | payment_issue | refund_exchange | other",
  "urgency": "low | medium | high",
  "needs_clarification": true,
  "route_to": "order_ops | shipping_ops | billing_ops | returns_ops | human_support"
}
```

데이터셋 파일:
- [dataset.jsonl](./dataset.jsonl)

## 스키마 정의

### `intent`

- `order_change`: 주문 수정, 취소, 주소 변경, 옵션 변경
- `shipping_issue`: 출고, 배송 지연, 배송 누락, 배송 완료 오표시
- `payment_issue`: 결제 실패, 중복 결제, 청구 이상
- `refund_exchange`: 반품, 환불, 교환, 불량 접수
- `other`: 위로 단정하기 어렵거나 맥락이 부족한 경우

### `urgency`

- `low`: 일반 문의, 즉시 장애 아님
- `medium`: 처리가 필요하지만 긴급 장애/금전 리스크는 아님
- `high`: 결제 이상, 분실/오배송, 고객 불만 고조, 수동 확인이 시급함

### `needs_clarification`

- `true`: 현재 텍스트만으로 intent 또는 처리 방향을 단정하기 어려움
- `false`: 현재 정보만으로 1차 분류 가능

### `route_to`

- `order_ops`: 주문/수정 담당
- `shipping_ops`: 배송 담당
- `billing_ops`: 결제/청구 담당
- `returns_ops`: 환불/교환 담당
- `human_support`: 맥락 부족, 다부서 이슈, 에스컬레이션 필요

## 구현 요구사항

### 필수

1. SDK를 사용해 LLM을 직접 호출할 것
2. `system` 메시지와 `user` 메시지를 분리할 것
3. `model`, `temperature`, `max_tokens`를 코드에 명시할 것
4. prompt v1, prompt v2 두 버전을 실험할 것
5. 응답을 JSON으로 파싱하고 코드에서 검증할 것
6. 12건 전체에 대해 결과를 저장할 것
7. v1과 v2의 결과를 비교한 짧은 리포트를 작성할 것

### 권장

- OpenAI 또는 OpenAI-compatible endpoint 사용
- Python 사용
- `pydantic` 또는 직접 작성한 검증 로직 사용

### 금지

- ChatGPT/Claude 웹 UI만으로 수행

# 정리
1. customer_message를 LLM에 넣는다.
2. 모델이 아래 4개 필드를 가진 JSON을 반환하게 한다.
  - intent
  - urgency
  - needs_clarification
  - route_to
3. 그 결과를 dataset.jsonl의 expected_output과 비교한다.
4. v1, v2 프롬프트별로 파싱 성공률과 exact match를 본다.
