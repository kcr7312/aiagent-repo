# 6주차 실습 과제: AI Agent 설계서 작성

## 배경

5주차까지는 LLM을 "한 번에 잘 쓰는 법"을 다뤘습니다. 이번 주는 다릅니다. LLM이 스스로 다음 행동을 결정하는 AI Agent를 직접 설계합니다. 구현과 평가는 이후 주차로 이어지므로, 이번 주의 산출물은 설계 문서 하나입니다.

Agent는 시작하자마자 코드를 쓰면 거의 망가집니다. "이 문제에 Agent가 필요한지", "어떤 Tool을 어떻게 쓸지", "언제 멈춰야 하는지"를 먼저 글로 결정해야 구현이 휘청이지 않습니다. 이번 과제의 목적은 그 결정 과정을 한 번 글로 통과시켜보는 것입니다.

### 이전 주차와의 연결

| 주차 | 이번 주와의 연결 |
|------|------------------|
| 2주차 | ReAct Thought 설계는 프롬프트 엔지니어링의 연장 |
| 3주차 | Golden Dataset의 `expected_answer` → 성공 판정 체크리스트 작성 감각 |
| 4주차 | RAG retriever → Agent의 Tool 하나로 재해석 |
| 5주차 | Ragas 평가(최종 응답) → 다음 주 Agent Eval(trajectory) 확장 |

## 과제 목표

본인이 만들고 싶은 AI Agent 하나를 고르고, 아래 템플릿을 채운 설계서 1개를 제출합니다. 도메인은 완전 자유입니다.

## 주제 선정 가이드

"Agent가 필요한 문제인가?"를 먼저 통과해야 합니다. 다음 두 조건을 모두 만족해야 Agent 후보입니다.

1. 사용자 요청에 따라 실행 경로가 달라진다 (고정 경로면 Workflow)
2. Tool을 2개 이상, 상황마다 다른 조합으로 호출해야 한다

판단이 애매하면 "같은 기능을 고정 함수로 10줄 안에 구현할 수 있는가?"를 자문합니다. 가능하면 Agent가 아닙니다.

OK 예시와 NG 예시:

| 분류 | 예시 | 이유 |
|------|------|------|
| OK | 출장 플래너 — 일정·숙소·항공·환율·날씨 tool을 요청에 따라 다르게 조합 | 스텝 수와 tool 순서가 매번 달라짐 |
| OK | 중고차 구매 어시스턴트 — 시세·매물·정비이력·세금 계산 tool 조합 | 사용자 의도에 따라 tool 선택 달라짐 |
| OK | 맛집 추천 — 검색·리뷰 요약·지도·영업시간 tool 조합 | 제약(예산·거리·영업시간)에 따라 경로 분기 |
| NG | 특정 PDF 하나에서 질문에 답하기 | RAG로 충분. 경로 고정 |
| NG | 텍스트 → JSON 변환 | 단일 LLM 호출 |
| NG | "A 하고 B 해" 고정 순서 태스크 | Workflow로 충분 |

Workflow 5패턴(Prompt chaining / Routing / Parallelization / Orchestrator-workers / Evaluator-optimizer) 정의는 `README.md` 참고.

## 제출물

- `design.md` 하나 — 아래 템플릿의 9개 섹션을 순서대로 채운 설계 문서
- (선택) `examples/` — Tool이 호출할 실제 API 응답이나 데이터셋 샘플 파일(JSON·CSV 등) 1~2건

## 설계서 템플릿

아래 9개 섹션을 이 순서로 전부 채웁니다. 각 섹션 끝의 "쓰는 요령"을 참고.

### 1. 개요·목적

- 해결하려는 문제 (1~2문장)
- 타깃 사용자 한 줄 요약
- 왜 Agent여야 하는가 — 단일 LLM 호출·RAG·Workflow로 안 되는 이유 1~2문단

쓰는 요령: "Agent가 필요한 이유"는 "요청마다 경로가 달라지는 구체 예시"로 증명합니다. 추상적으로 "지능적이라서"는 탈락.

### 2. 사용자 시나리오

- Persona 1명 (이름·역할·목적)
- 대표 요청 2~3개 (자연어 원문)
- 각 요청이 단일 Tool 한 번으로 끝나지 않는 이유 (요청별 한 줄)

쓰는 요령: 요청 3개는 서로 다른 경로를 타도록 고릅니다. 셋 다 같은 tool 순서로 풀린다면 Workflow입니다.

### 3. 기능 요구사항

- Must-have 3~5개
- Nice-to-have 2~3개

각 항목은 "어떤 입력을 받아 어떤 출력을 낸다"로 씁니다. 구현 방법이 아니라 행동을 적습니다.

### 4. Agent 패턴 선택과 근거

패턴별 루프 구조·강점·약점은 `README.md`의 Agent 패턴 표 참고.

- 선택한 패턴: ReAct / Reflection·Reflexion / Plan-and-Execute / Multi-Agent 중 하나 (또는 혼합)
- 선택 근거 2~3줄 — 왜 이 문제에 이 패턴이 맞는가
- 간단한 루프 구조도 (의사코드 또는 순서 설명 5~10줄)

쓰는 요령: 채점은 없습니다. 단, 근거가 "가장 유명해서"면 재검토. "요청 스텝 수가 예측 가능한가 / 자기검증이 필요한가 / 역할 분업이 필요한가"로 선택합니다.

### 5. 동작 명세

전통적 SRS의 "입력·출력·흐름"에 Agent 고유 항목(종료 조건·실패 흐름)을 더한 것입니다.

- 입력 스키마 — 사용자 입력 형태 (자연어 문자열? 구조화 필드 포함?)
- 출력 스키마 — 최종 응답 포맷 (자연어? JSON? 둘 다?)
- 정상 흐름 — 시나리오 1개를 골라 Thought/Action/Observation trace 예시 (5~10 스텝)
- 예외 흐름 — Tool 실패·답변 불가·권한 부족 시 동작
- 종료 조건 — max steps, stop signal, 반복 탐지 기준

쓰는 요령: 정상 흐름 예시는 "모범 답안 trajectory"입니다. 이 흐름이 8번에 나올 성공 판정 체크리스트의 기준이 됩니다.

### 6. Tool 명세

| Tool 이름 | 목적(1줄) | 입력 스키마 | 출력 스키마 | 실패 시 반환 | 사용 조건 |
|----------|----------|----------|----------|------------|----------|
| ... | ... | ... | ... | ... | ... |

Tool 개수는 2~5개를 권장합니다. 너무 많으면 LLM이 고르다가 실패합니다.

쓰는 요령:

- 설명은 "LLM이 이것만 보고 언제 쓸지 판단할 수 있는가"를 기준으로 씁니다
- 실패는 예외 throw 대신 `{error: "...", detail: "..."}` 같은 결과 객체로 돌려줍니다 (LLM이 읽고 다음 행동을 고를 수 있게)
- 파괴적 동작(삭제·결제 등)은 확인 파라미터를 넣습니다 (`confirm=True`)

### 7. 데이터셋

각 Tool이 실제로 어떤 데이터를 주고받는지 구체 예시를 붙입니다.

- Tool별 데이터 출처 (API URL·데이터셋 이름·파일 경로)
- 실제 응답 샘플 1~2건 (JSON·CSV의 실제 값)
- 데이터 특성 요약 (필드 몇 개, 업데이트 주기, 인증 필요 여부)
- 실제 데이터가 없으면 Mock 명시 (스키마는 동일하게)

데이터셋 탐색 출발점:

- [public-apis-4Kr](https://github.com/yybmion/public-apis-4Kr) — 한국 공공·민간 API 모음
- [GLOBAL_PUBLIC_APIS_KR.md](https://github.com/yybmion/public-apis-4Kr/blob/main/GLOBAL_PUBLIC_APIS_KR.md) — 해외 공개 API 한국어 정리
- 보조: [Hugging Face Datasets](https://huggingface.co/datasets), [공공데이터포털](https://www.data.go.kr), [서울열린데이터광장](https://data.seoul.go.kr)

쓰는 요령: 실제 호출 한 번 해보고 응답 JSON을 그대로 붙이는 것이 가장 빠릅니다. 인증·쿼터 이슈가 있으면 mocking 규칙(응답 스키마·샘플 값 몇 개)을 명시합니다.

### 8. 성공 판정 기준

"이 Agent가 잘 동작한다"를 체크리스트 3~5개로 선언합니다. 구현·평가는 다음 주 일이지만, 기준을 먼저 못 쓰면 나중에 평가 셋을 못 만듭니다.

항목 예시:

- "요청 X에 대해 Tool A → Tool B 순서로 호출한다"
- "Tool 호출 실패 시 재시도 또는 다른 tool로 fallback"
- "최종 응답에 Tool 결과의 특정 필드가 포함된다"
- "6 step 이내에 종료한다"
- "같은 요청을 두 번 넣었을 때 경로가 과도하게 달라지지 않는다"

쓰는 요령: 각 항목은 "예/아니오로 판정 가능"해야 합니다. "똑똑하게 판단한다" 같은 모호한 기준은 탈락. 다음 주차 Agent Eval에서 다룰 final/step/trajectory 용어는 `README.md`의 설계 축 표 참고.

### 9. 제약·확장

- 현재 설계의 한계 (단일 Agent 기준으로 풀리지 않거나 어색한 지점)
- Multi Agent로 확장한다면 어디를 쪼갤지 (Orchestrator + Worker 역할 분리 후보)
- 장기 상태·메모리가 필요해지는 시나리오 (7주차 심화 연결)

쓰는 요령: 이 섹션은 7주차 Multi Agent로 넘어가는 예고편입니다. 지금 설계가 부족해서 채우는 섹션이 아니라, "확장 지점을 미리 식별"하는 섹션입니다.

## 자가 점검 체크리스트

채점은 하지 않습니다. 다만 제출 전에 아래 5개를 스스로 점검합니다.

1. RAG만으로 풀 수 있는 문제가 아닌가 (경로가 매 요청마다 달라지는가)
2. Tool 설명이 LLM 관점에서 "언제 쓸지"를 결정할 수 있을 만큼 구체적인가
3. 종료 조건과 실패 흐름이 둘 다 명시됐는가
4. 데이터셋이 실재하거나(출처 URL) Mock 규격이 스키마 수준으로 뚜렷한가
5. 제약·확장 섹션에 "다음 주 Multi Agent로 쪼갤 지점"이 최소 한 개 식별됐는가

## 참고: 예시 Agent 주제 3가지 (고르지 말고 감만 잡기)

- 중고차 매입 어시스턴트 — 시세 tool, 매물 tool, 정비 이력 tool, 세금 계산 tool
- 주말 플래너 — 날씨 tool, 장소 검색 tool, 교통 tool, 일정 충돌 검사 tool
- 개인 주식 리서처 — 시세 tool, 공시 검색 tool, 뉴스 요약 tool, 환율 tool

위 셋은 예시입니다. 그대로 베끼면 사고 연습이 되지 않으니 본인의 실제 문제를 고르길 권장합니다.
